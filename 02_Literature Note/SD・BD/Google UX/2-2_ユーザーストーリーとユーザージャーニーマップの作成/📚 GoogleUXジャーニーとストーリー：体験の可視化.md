### ユーザージャーニーマップ

- [[ジャーニーの書き方]]
- [[ジャーニーマップの役割]]

**ジャーニーマップとは**:
人々の行動を時系列に沿ってマッピングしたもの。顧客の選択や決断に至る道を可視化する。

**ジャーニーマップに含まれる要素**:
1. **アクター（Actor）**: 誰のジャーニーか？（ペルソナ）
2. **シナリオ（Scenario）**: 何を達成しようとしているか？
3. **ステージ（Stages）**: プロセスの各段階
4. **アクション（Actions）**: 各段階での具体的な行動
5. **思考（Thoughts）**: 各段階での考え
6. **感情（Emotions）**: 各段階での感情（グラフ化することも）
7. **タッチポイント（Touchpoints）**: ユーザーと製品/サービスの接点
8. **機会（Opportunities）**: 改善の機会、ペインポイント

**ジャーニーマップの役割**:
- ユーザー体験の全体像を可視化
- チーム内で共通理解を作る
- ペインポイントを特定
- 改善の機会を発見
- 優先順位付けの根拠

### ユーザー定義文とジョブ定義文

- [[ユーザー定義文とジョブ定義文]]

**ユーザー定義文（User Story）**:

```
As a [WHO：type of user], 
I want to [WHAT：action] 
so that [WHY：benefit].
```

**特徴**:
- フォーカスは**行動**
- ユーザーロールが出発点
- 開発要件化が目的
- アクセスしやすく記述しやすいので、関係者間の目線合わせを容易にする
- ジョブから落ちてくる解決シナリオの断片（1段階具体）

**例**:
> As a **working parent**, 
> I want to **find reliable childcare quickly** 
> so that **I can go to work without worry**.

---

**ジョブ定義文（Job Statement）**:

```
When [SITUATION], 
I want to [MOTIVATION], 
so I can [EXPECTED OUTCOME].
```

**特徴**:
- ユーザーロールは排除していて、**状況性**に依存する
- より抽象的で、新たな課題発見やコンセプト創出の核になる
- 1つのジョブから、無数のアイデアが出てくる

**ユーザー定義文との違い**:

| 観点 | ユーザー定義文 | ジョブ定義文 |
|------|--------------|------------|
| **出発点** | ユーザーロール（WHO） | 状況（WHEN） |
| **抽象度** | 具体的（解決シナリオ） | 抽象的（進歩の本質） |
| **目的** | 開発要件化、目線合わせ | 課題発見、コンセプト創出 |
| **アイデア** | 1対1対応 | 1つのジョブ→無数のアイデア |

**例（ジョブ定義文）**:
> When **I'm commuting to work alone in my car**, 
> I want to **make the time pass quickly without getting bored**, 
> so I can **arrive at work in a good mood**.

→ このジョブから、ミルクシェイク、Podcast、オーディオブックなど、様々なソリューションが生まれる

## 💡 実践への応用

### ジャーニーマップ作成のステップ

1. **アクターを定義する**
   - [ ] ペルソナは明確か？
   - [ ] 複数のペルソナがいる場合、どのペルソナのジャーニーか？

2. **シナリオを設定する**
   - [ ] 何を達成しようとしているか？
   - [ ] どんな状況か？

3. **ステージを分解する**
   - [ ] 体験の開始から終了までを段階に分ける
   - [ ] 例：認知→検討→購入→使用→サポート

4. **各ステージで記録する**
   - [ ] アクション：何をしているか？
   - [ ] 思考：何を考えているか？
   - [ ] 感情：どう感じているか？（高い/低い）
   - [ ] タッチポイント：何と接触しているか？

5. **機会を特定する**
   - [ ] どこでペインが発生しているか？
   - [ ] どこで改善の機会があるか？
   - [ ] 優先順位は？

### ユーザーストーリー vs ジョブ定義文の使い分け

**ユーザーストーリーを使う場面**:
- 開発チームとの要件共有
- スプリント計画
- 具体的な機能の優先順位付け
- 関係者間の目線合わせ

**ジョブ定義文を使う場面**:
- 新規プロダクト/サービスのコンセプト創出
- 市場機会の探索
- イノベーションのアイデア発想
- ユーザーの本質的ニーズの理解

**実践のコツ**:
1. 最初はジョブ定義文で本質を捉える
2. 次にユーザーストーリーで具体化する
3. ユーザーストーリーは開発チームと共有
4. ジョブ定義文は定期的に見直して、ソリューションが本質からずれていないか確認

## 🎯 ユーザーストーリーとジョブの対応例

**ジョブ定義文**:
> When I'm commuting alone, I want to make time pass quickly, so I can arrive in a good mood.

**↓ 具体化**

**ユーザーストーリー（例1）**:
> As a **commuter**, I want to **listen to podcasts**, so that **I can learn something during my commute**.

**ユーザーストーリー（例2）**:
> As a **commuter**, I want to **drink a milkshake**, so that **I can enjoy something tasty and filling**.

**ユーザーストーリー（例3）**:
> As a **commuter**, I want to **play audiobooks**, so that **I can enjoy stories hands-free**.

→ 1つのジョブから、複数のユーザーストーリー（ソリューション）が生まれる

## ✨ このセクションから得られる最も重要な学び

1. **ジャーニーマップは体験の可視化ツール**: 時系列で行動・思考・感情を整理
2. **ユーザーストーリーは開発のため**: 具体的な機能を要件化
3. **ジョブ定義文は発見のため**: 本質的なニーズを抽象化
4. **1つのジョブ→無数のソリューション**: ジョブから考えると選択肢が広がる
5. **感情の波に注目**: ジャーニーマップの感情の高低が改善の機会
