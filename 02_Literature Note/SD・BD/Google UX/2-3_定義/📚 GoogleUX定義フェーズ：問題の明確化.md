### Problem Statement

- [[Problem Statementの定義]]
- [[Problem Statementとジョブ定義文の違い]]

**Problem Statementとは**:
ユーザーが抱える問題を明確に記述したもの。デザインの方向性を決める羅針盤となる。

**フォーマット**:
```
[USER] needs [NEED] because [INSIGHT].
```

**例**:
> **Working parents** need **a way to find reliable childcare quickly** because **they have limited time and high anxiety about their children's safety**.

**良いProblem Statementの条件**:
1. **ユーザー中心**: 誰のための問題か明確
2. **ニーズが明確**: 何を必要としているか具体的
3. **インサイトがある**: なぜその問題が重要か示す
4. **ソリューションを含まない**: 「何を」ではなく「なぜ」に焦点

---

**Problem Statementとジョブ定義文の違い**:

| 観点 | Problem Statement | ジョブ定義文 |
|------|------------------|------------|
| **フォーカス** | 問題の明確化 | 達成したい進歩 |
| **構造** | USER / NEED / INSIGHT | SITUATION / MOTIVATION / OUTCOME |
| **目的** | デザインの方向性を定める | コンセプト創出、アイデア発想 |
| **使用タイミング** | 定義フェーズ（リサーチ後） | ブレインストーム、コンセプト段階 |

**使い分け**:
- **Problem Statement**: 「この問題を解決する」という設計の起点
- **ジョブ定義文**: 「この進歩を達成する」という本質の理解

両方を組み合わせることで、問題の本質と解決の方向性が明確になる。

関連: [[ジョブ理論の本質と実践]]、[[インサイト抽出とHMW]]

### 潜在ペインの発見

- [[潜在的なペインの見つけ方]]
- [[エンパシーマップのSAYとDOの差が、潜在ニーズ特定の１つのコツ]]
- [[潜在ペインは、社会的に整えられたSAYを指摘し、実態として生々しい行動との矛盾を特定する行為なので、インタビュー中に深追いするとインタビュイーの心理的安全性を損なわせる行為に直結する。だから余計に矛盾を突っ込みすぎてはいけないんだな。]]

**潜在ペインとは**:
ユーザー自身が認識していない、または言語化できていない痛み・課題。

**見つけ方**:

1. **SAYとDOの矛盾に注目**
   - **SAY（言うこと）**: インタビューでの発言、建前、社会的に整えられた表現
   - **DO（すること）**: 実際の行動、本音、生々しい実態
   - **ギャップ**: SAYとDOの差が潜在ニーズを示す

   **例**:
   - SAY: 「健康的な食事を心がけている」
   - DO: コンビニ弁当を週5で買っている
   - **潜在ペイン**: 健康的な食事を簡単に手に入れる方法がない

2. **「マインドリーディング」の必要性**
   - ユーザーは自分の問題を正確に表現することに抵抗がある
   - 暗黙のペインは存在し、それを特定することは重要なタスク
   - しかし、インタビュー中に深追いしすぎてはいけない

3. **心理的安全性の配慮**
   - 潜在ペインは社会的に整えられたSAYを指摘する行為
   - 実態として生々しい行動との矛盾を特定する行為
   - インタビュー中に深追いすると、心理的安全性を損なう
   - **だから、矛盾に気づいても、その場で強く突っ込みすぎてはいけない**

関連: [[インタビュー技術の本質]]、[[観察の技術と哲学：深く見る力の育て方]]

### インタビューにおける矛盾の扱い

- [[インタビューのその場で矛盾に気づけるのか？]]

**矛盾に気づくタイミング**:

1. **その場で気づく場合**
   - 反射神経で「あれ？」と思う瞬間
   - しかし、強く突っ込みすぎない（心理的安全性）
   - 「もう少し詳しく教えてください」と柔らかく聞く

2. **事後分析で気づく場合**
   - 共感マップを作成しながら気づく
   - 録音を文字起こししながら気づく
   - 複数のインタビューを比較して気づく

**実践のコツ**:
- インタビュー中は、矛盾に気づいても記録だけにとどめる
- 「なぜそうしたんですか？」ではなく、「そのとき何を考えていましたか？」
- 事後分析で、ゆっくりと矛盾を解釈する

関連: [[インタビュー技術の本質]]（理由を強制しない）

### 具体行動の引き出し方

- [[語りが少ない・抽象発言が苦手なインタビュイーには、具体行動や選択を引き出すべし]]
- [[数年前の行動に対するインタビュー：過去行動を直接聞くのではなく、"記憶に残る瞬間"を聞く]]

**語りが少ないインタビュイーへの対応**:

1. **具体行動を聞く**
   - 「どう思いますか？」→ ❌（抽象的すぎる）
   - 「最後にそれをしたのはいつですか？」→ ✅
   - 「そのとき何をしましたか？」→ ✅

2. **選択を聞く**
   - 「どんな選択肢を考えましたか？」
   - 「なぜそれを選んだのですか？」
   - 「他にどんな方法がありましたか？」

3. **記憶に残る瞬間を聞く**
   - 「印象に残っている出来事はありますか？」
   - 「一番困ったのはどんな時ですか？」
   - 「嬉しかったのはどんな時ですか？」

**数年前の行動に対するインタビュー**:
- 過去行動を直接聞くのではなく、**"記憶に残る瞬間"を聞く**
- 人は詳細を忘れるが、感情が動いた瞬間は覚えている
- 「3年前に何をしましたか？」→ ❌
- 「3年間で一番困ったことは？」→ ✅

関連: [[インタビュー技術の本質]]（時系列インタビュー）

## 💡 実践への応用

### Problem Statement作成のステップ

1. **インタビュー・観察を実施する**
   - [ ] 共感マップを作成したか？
   - [ ] SAYとDOの矛盾を特定したか？
   - [ ] ペインポイントを分類したか？

2. **ユーザーを特定する（WHO）**
   - [ ] ペルソナは明確か？
   - [ ] どのユーザーセグメントか？

3. **ニーズを明確にする（NEED）**
   - [ ] 何を必要としているか？
   - [ ] 「〜する方法」で表現できるか？
   - [ ] ソリューションを含んでいないか？（含んでいたら修正）

4. **インサイトを加える（INSIGHT）**
   - [ ] なぜその問題が重要か？
   - [ ] SAYとDOの矛盾から導かれたか？
   - [ ] 新規性があるか？

5. **フォーマットに当てはめる**
   ```
   [USER] needs [NEED] because [INSIGHT].
   ```

6. **検証する**
   - [ ] ユーザー中心か？
   - [ ] 具体的か？
   - [ ] ソリューションを含んでいないか？
   - [ ] チームで合意できるか？

### 潜在ペイン発見のチェックリスト

#### インタビュー中

- [ ] 「もう少し詳しく」を使っているか？
- [ ] 心理的安全性を保てているか？

#### 事後分析
- [ ] 共感マップでSAYとDOを整理したか？
- [ ] SAYとDOのギャップを特定したか？
- [ ] そのギャップの背後にある理由を推論したか？
- [ ] 推論はデータに基づいているか？（仮説に引っ張られていないか？）

## 📊 SAYとDOの矛盾分析フレームワーク

```
┌──────────────────────────────────────────┐
│         SAY（言うこと）                    │
│                                          │
│  「健康的な食事を心がけている」             │
│  「時間をかけて料理したい」                │
│  「栄養バランスを考えている」              │
│                                          │
└──────────────────────────────────────────┘
                    ↕
          【ギャップ = 潜在ニーズ】
                    ↕
┌──────────────────────────────────────────┐
│         DO（すること）                     │
│                                          │
│  コンビニ弁当を週5で買っている             │
│  電子レンジで温めるだけ                   │
│  栄養成分表示を見ていない                 │
│                                          │
└──────────────────────────────────────────┘

↓ 解釈

【潜在ペイン】:
「健康的な食事を簡単に手に入れる方法がない」
「料理する時間とエネルギーがない」
「栄養を気にしつつも、利便性を優先せざるを得ない」
```

## 🎯 Problem Statementの例

**悪い例**:
> Users need **a mobile app** because they want convenience.

- ❌ ソリューション（mobile app）を含んでいる
- ❌ インサイトが薄い（「convenience」は表面的）

**良い例**:
> **Working parents** need **a way to quickly find and book reliable childcare** because **they experience high anxiety when their regular childcare falls through and they have limited time to find alternatives**.

- ✅ ユーザーが明確（Working parents）
- ✅ ニーズが具体的（quickly find and book reliable childcare）
- ✅ インサイトが深い（高い不安、時間の制約、代替手段の不足）
- ✅ ソリューションを含んでいない（「mobile app」とは言っていない）

---

## ✨ このセクションから得られる最も重要な学び

1. **Problem Statementはデザインの羅針盤**: ユーザー・ニーズ・インサイトで構成
2. **SAYとDOの矛盾が潜在ニーズ**: 言っていることと実際の行動の差に注目
3. **矛盾に気づいても深追いしない**: 心理的安全性を保つことが最優先
4. **事後分析でゆっくり解釈**: インタビュー中は記録、後でじっくり考える
5. **具体行動と記憶に残る瞬間**: 抽象的な質問より、具体的な出来事を聞く
