#留学生就労支援

## エグゼクティブサマリー

How Might We?（HMW）は、「どうすれば私たちは〜できそうか？」という問いの形式を用いて、課題を解決策を生み出すための機会へと転換する、デザイン思考における重要な手法である。このアプローチの目的は、性急に単一の解決策に飛びつくのではなく、多様なアイデアを探求するための創造的な土台を築くことにある。HMWは、リサーチから得られたユーザーのニーズや課題（インサイト）を基に、意図的に解決策の方向性を設定し、アイデア出し（アイディエーション）のフェーズを効果的に促進する。

質の高いHMWステートメントは、具体的すぎず、かつ抽象的すぎない「ちょうど良い」レベルの抽象度を持つことが不可欠である。具体的すぎる問いはアイデアの幅を狭め、抽象的すぎる問いは方向性を見失わせる。スタンフォード大学d.schoolなどが提唱する多様なフレームワーク（例：「良い面を伸ばす」「前提を疑う」）を活用することで、多角的な視点から質の高い問いを数多く生み出すことが可能となる。このプロセスは、チームの共同作業を促し、課題解決に向けた思考の密度と質を向上させるための強力なツールとして機能する。

## 1. HMWの定義と目的

How Might We?（HMW）は、デザイン思考のアクティビティであり、問題をデザインの機会へと変換するために使用される。日本語では「どうすれば私たちは〜できそうか？」と訳され、課題解決に向けたアイデア出し（アイディエーション）のフレームとなる問いを設定する手法である。この手法は1970年代にProcter & Gamble社によって導入され、後にデザインファームのIDEO社が採用したことで広く知られるようになった。

HMWのフレーズは、以下の3つの要素で構成されている。

|   |   |   |
|---|---|---|
|要素|意味|解説|
|**How**|**方法の探求**|単一の解決策に固執せず、多数のアイデアを探求することを促す。具体的な方法を問うことで、実行可能なアクションを志向させる。|
|**Might**|**可能性の示唆**|提案されるアイデアが唯一絶対の解決策ではなく、あくまで「可能性のある」解決策の一つであることを強調する。「can」や「should」ではなく「might」を使うことで、判断を保留し、より自由で創造的な発想を促す。|
|**We**|**協働の示唆**|課題解決が個人ではなく、チームによる共同作業であることを示唆する。|

HMWの主な目的は、ユーザーインタビューなどのリサーチから得られたニーズや課題に対し、そのまま解決策を考えるのではなく、一度「問い」の形にリフレーミングすることである。これにより、プロジェクトが取り組むべき課題の方向性が明確になり、アイデアの発散が容易になる。

## 2. HMWの重要性と役割

デザイン思考のプロセスにおいて、HMWはリサーチから得られた洞察（インサイト）を具体的なアイデアに繋ぐための重要な架け橋となる。

### 2.1. アイディエーションの失速防止

デザイン思考のワークショップでは、共感マップやジャーニーマップを用いてユーザー理解を深めた後、具体的なアイデアを考えるアイディエーションのフェーズで勢いが落ちることが多い。これは、分析からアイデア創出への間に思考のジャンプがあるためである。HMWは、解決すべき課題を明確に定義することでこのギャップを埋め、アイディエーションの方向性を定める羅針盤として機能する。

### 2.2. 「ちょうど良い」制約の設定

HMWは、アイデアを発散させるための「適度な制約」を設ける役割を持つ。制約がなければアイデアは無秩序に発散し収集がつかなくなるが、制約が厳しすぎると創造性が損なわれる。HMWによって解決策の方向性を制限することで、チームはその分野に集中して密度の濃いアイデア出しを行うことが可能になる。

### 2.3. 問題のリフレーミング

HMWは、課題を異なる視点から捉え直す（リフレーミングする）ことを促す。これにより、創造性が引き出され、ユーザーが直面している問題に対して多種多様な解決策を生み出す新しい視点が得られる。

## 3. 質の高いHMWステートメントの作成プロセス

質の高いHMWステートメントを作成するには、体系的なステップを踏むことが推奨される。前提として、ユーザーリサーチが完了し、ユーザーのペインポイントや課題感が洗い出されている必要がある。

#### STEP 1: 問題の定義（ニーズステートメントの作成）

リサーチ結果を基に、まず解決すべき問題を定義する。以下のフレームワークが有効である。 `**[ユーザー]**` **は** `**[要望・ニーズ]**` **をする必要がある。なぜなら** `**[リサーチの結果・洞察]**` **だからだ。**

- **例：** 「第一子を控えている親（ユーザー）は貯蓄投資口座を開く必要がある（ニーズ）。なぜなら彼らは子供の将来の教育計画を立てたいが、たくさんの選択肢と口座開設の方法に圧倒されているからだ（洞察）。」

#### STEP 2: 問題の要素分解

定義した問題（ニーズステートメント）はまだ大きすぎることが多いため、実行可能な小さな断片に分解する。

- **例：**
    - 選択を手助けする
    - 手続きを手助けする
    - 投資型貯蓄の意味を理解させる

#### STEP 3: できるだけ多くのHMWを考える

分解した要素を基に、HMWステートメントのブレインストーミングを行う。以下のフレームワークが役立つ。 **どのようにして** `**[あるべきユーザー体験]**` **を実現し、** `**[ユーザー]**` **が** `**[期待される効果]**` **を得られるようにできるか？**

- **例：** 「第一子を控える親（ユーザー）が自分の選択に自信を持てるようにする（期待される効果）ためには、どのようにして投資型貯蓄の選択をより簡単にする（あるべきユーザー体験）ことができるか？」
- **注意点：** この段階では、手段や方法（例：モバイルアプリ、AI）を具体的に定義しないことが重要である。あくまで「体験レベル」で考える。

#### STEP 4: （必要に応じて）制約を設ける

制約を加えることで、より面白いアイデアが生まれる可能性がある。特に、矛盾やパラドックスを含むHMWは優れたアイデアの源泉となり得る。

- **例：** 「第一子を控える親（ユーザー）が **いかなる金融商品の理解度に関わらず（制約）** 自分の選択に自信を持てるようにする（期待される効果）ためには、どのようにして投資型貯蓄の選択をより簡単にする（あるべきユーザー体験）ことができるか？」

#### STEP 5: 良いHMWステートメントを選ぶ

生成した多数のHMWの中から、チームで議論し、最も筋が良く、多様なソリューションが生まれそうな「良い問い」を最大3つ程度に絞り込む。

## 4. 良いHMWの特性：「ちょうど良い」抽象度

HMWが効果的に機能するためには、その抽象度が「ちょうど良い」レベルにあることが極めて重要である。

- **抽象的すぎるHMWの例:**
    - `どうしたらインドの公衆トイレをリデザインできるか？`
    - `どうすればチケット販売をより良くできるか？`
    - **問題点:** 方向性が広すぎて、アイデアが発散しすぎてしまう。解決策のアイデアを思いつくのに十分な指針を提供しない。
- **具体的すぎるHMWの例:**
    - `どのようにしたら清潔で安全で人を迎え入れるようなインドの公衆トイレのドアノブをリデザインできるか？`
    - `ペットボトルを再利用して、飛行機のシートベルトを作るにはどうしたらいいか？`
    - **問題点:** すでに解決策（ドアノブ、シートベルト）が定義されており、アイデアの幅が極端に狭まってしまう。
- **ちょうど良いHMWの例:**
    - `どのようにしたらインドの公衆トイレの安全性という概念を生み出すことができるか？`
    - `どうすれば我々は、アイスクリームをより持ち運びやすくできるか？`
    - **特徴:** 複数の解決策を許容する広さを持ちながら、解決策を絞るのに十分な方向性も示している。ある講師は、このレベル感を「モネの絵くらいの抽象度を目指せ」と表現した。

## 5. HMW発想のためのフレームワークとヒント

ブレインストーミングで行き詰まった際に、多角的な視点からHMWを生成するためのフレームワークがスタンフォード大学d.schoolなどから提唱されている。以下にその主要なアプローチをまとめる。

|   |   |   |
|---|---|---|
|アプローチ|説明|例（空港での待ち時間という課題に対して）|
|**良いところを伸ばす (Amp up the good)**|問題の中のポジティブな要素を解決策として活用する。|`どうすれば、子供たちのエネルギーで他の乗客を楽しませることができるだろうか？`|
|**悪いところをなくす (Remove the bad)**|問題のネガティブな部分を完全に取り除く方法を考える。|`どうすれば、子供たちを他の乗客から遠ざけることができるだろうか？`|
|**逆を考える (Explore the opposite)**|問題の逆を解決する方法を考える。|`どうすれば、待ち時間を旅行で最も面白い部分にできるだろうか？`|
|**前提を疑う (Question an assumption)**|必要だと思われるプロセスを削除・変更する。|`どうすれば、空港での待ち時間を完全に無くすことができるだろうか？`|
|**形容詞を変えてみる (Go after the adjective)**|ネガティブな形容詞をポジティブな言葉に変える。|`どうすれば、「つまらない」待ち時間を、「快適な」待ち時間に変えられるだろうか？`|
|**他のリソースを見つける (Identify unexpected resources)**|問題文にないリソースによって問題がどう解決されるか考える。|`どうすれば、他の乗客の自由な時間を活用できるだろうか？`|
|**アナロジー（類推）を生み出す (Create an analogy)**|ユーザー体験を他の体験と比較する方法を考える。|`どうすれば、空港を温泉や遊び場所のようにできるだろうか？`|
|**課題に正面からぶつかる (Apply POV to the challenge)**|着眼点（インサイト）を課題に直接適用する。|`どうすれば、空港を子供たちが行きたくなる場所にできるだろうか？`|
|**現状を打破する (Change a status quo)**|プロセスを完全に変える方法を考える。|`どうすれば、はしゃいで騒ぐ子供たちを静かにできるだろうか？`|
|**問題を分割する (Break the POV into pieces)**|視点をバラバラにして、個別の問題として考える。|`どうすれば子供たちを楽しませられるか？` `どうすれば母親を落ち着かせられるか？`|

## 6. HMW作成のベストプラクティス

- **Be broad（広くあれ）:** 良いHMWは複数の解決策を許容する。
- **But don’t be too broad（ただし広すぎるな）:** 包括的でありながら、解決策を絞るのに十分な幅を持たせる。
- **Make multiple drafts（複数の下書きを作る）:** HMWが有用な解決策を導かない場合は、ためらわずに質問を変更する。
- **Be creative（創造的であれ）:** HMWの作成は想像力豊かで楽しいプロセスである。前述のフレームワークを活用する。
- **Write as many HMWs as you can（できるだけ多く書く）:** HMWが多ければ多いほど、より多くの解決策を思いつくことができる。

## 7. 具体例

#### ケース1：D2C植物ストア「The Sill」

- **インサイト/課題:**
    1. オンラインが主であるため、購入前に植物の健康状態を直接確認できない。
    2. 店員に手入れの方法を気軽に質問できない。
- **悪いHMWの例:** `どのようにしたらThe Sillの宅配を改善できるか？` (教授から「今まで見た中で最悪のHMW」と評された)
- **良いHMWの例:**
    - `どのようにしたら実際の店舗で買い物をしている時の体験がオンラインで実現できるか？`
    - `どのようにしたら顧客が植物を買う前に実際に商品を「見る」ことができるか？`

#### ケース2：遊園地のアトラクションの待ち時間

- **課題:** アトラクションの待ち時間が長く、来場者が退屈してしまう。
- **HMWのバリエーションと解決策の方向性:**
    - `どうすれば我々はアトラクションの待ち時間を減らせるだろうか？` → 稼働率向上などの運営効率化
    - `どうすれば我々はアトラクションの待ち時間を楽しんでもらえるだろうか？` → 待機列でのエンターテイメント提供（隠れキャラクター探しなど）
    - `どうすれば我々は待ち時間中にお客さんが他のアトラクションを楽しめるようにできるだろうか？` → ファストパスのようなシステムの導入

#### ケース3：ソフトウェア開発チームの課題

- **課題:** 新しい技術の習得が難しく、計画通りに作業が終わらなかった。
- **HMWのバリエーション:**
    - **良い面を伸ばす:** `どうすれば我々は、新しい技術を使いこなせるだろうか？`
    - **悪い面を除去する:** `どうすれば我々は、作業の見積もりを改善することができるだろうか？`
    - **反対を探す:** `どうすれば我々は、新しい技術を使わずに問題を解決できるだろうか？`
    - **他のリソースを活用する:** `どうすれば我々は、他のチームに相談して難しい作業を終わらせられるだろうか？`
