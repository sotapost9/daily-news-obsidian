## 🎯 現状分析の気づき

### EC分析.mdからの観察

**ポジティブな点：**
- 「Pokから探す」機能が優れている
- 不正ログイン検知・停止まで対応
- お勧めキャンペーンや新商品を全面的に出す

**課題・疑問点：**
- 特集とニュースの違いがわかりにくい
- グッズ、ゲーム、キャンペーンでセグメントが異なる可能性
- カテゴリーが多岐にわたり、構造が見えづらい
- 関連サイトが乱立している？

**示唆：**
- ユーザーセグメント（グッズ好き、ゲーマー、キャンペーン目当て）ごとに異なるジョブがある可能性
- 「Pok好き」という共通項だけで統一体験を作るのは難しい
- カテゴリー構造の見直しが必要かもしれない

### ポケセン分析.mdからの店舗の強み

**店舗体験の特徴：**
- 店舗ごとに代表キャラのモニュメント（「ポケセンに来た感」）
- 「ワクワク」をコンセプトにしている
- 誕生日特典（Pok受け取り、ついで買い誘発）
- 各店舗のコンセプト差別化（1日で5軒回る企画が成立）

**ECへの示唆：**
- オンラインで「ワクワク」をどう再現するか
- パーソナライゼーション（誕生日特典など）の強化
- 店舗との連携（OMO）の可能性

## 🎯 ジョブ理論：顧客の真のニーズを理解する

[[ジョブ理論の本質と実践]]

### ジョブ理論の核心

> 「顧客は製品を買うのではなく、達成したい『進歩』のために製品を『雇う』」

**ECサイトへの適用：**
- 何を（商品）ではなく、なぜ（ジョブ）を理解する
- 状況 × 進歩 × 障害 × 代替案で分析

### PokECのジョブ仮説（要検証）

#### ジョブ1：「推しPok」関連グッズ収集
- **状況**：新商品が出た時、Twitterで話題になった時
- **進歩**：推しPokに囲まれた生活、コレクションの充実感
- **障害**：在庫切れ、店舗が遠い、どこに何があるか分からない
- **代替案**：店舗に行く、メルカリで探す、諦める

#### ジョブ2：ギフト・プレゼント
- **状況**：誕生日、クリスマス、ちょっとしたお礼
- **進歩**：相手を喜ばせたい、センスの良さを示したい
- **障害**：相手の好きなPokが分からない、予算感が不明
- **代替案**：Amazonギフト券、他のキャラクターグッズ

#### ジョブ3：「自分へのご褒美」
- **状況**：仕事や勉強を頑張った後、ストレスが溜まった時
- **進歩**：気分転換、癒し、小さな幸福感
- **障害**：罪悪感（また買ってしまった）、予算オーバー
- **代替案**：他のEC、実店舗、我慢する

#### ジョブ4：ゲームプレイヤーとしての補完
- **状況**：新作ゲーム発売、イベント開催
- **進歩**：ゲーム体験の拡張、特典入手
- **障害**：予約を忘れた、売り切れ
- **代替案**：ゲームショップ、ダウンロード版

### ジョブを発見する5ステップ

[[デザインリサーチで使えるジョブインタビュー実践ガイド]]

**インタビュー質問設計：**

1. **進歩は何か**
   - 「最後にPokセンターオンラインで買い物をしたのはいつですか？」
   - 「そのとき、何を達成しようとしていましたか？」

2. **苦心している状況**
   - 「どんな文脈でそのジョブが立ち上がったのですか？」
   - 「繰り返し起きることですか？」

3. **阻む障害**
   - 「それを実現するために、何か試しましたか？」
   - 「何が邪魔をしていますか？」

4. **不完全な解決策**
   - 「今は、どのように対処していますか？」
   - 「10点満点で何点ですか？」

5. **より良い解決策の条件**
   - 「理想的な解決策は、どんなものですか？」
   - 「何を諦めてもいいと思いますか？」

### ジョブ理論の3つの側面

**機能的ジョブ：**
- 商品を探す、購入する、受け取る
- 在庫を確認する、比較する

**感情的ジョブ：**
- ワクワクする、癒される、満足感を得る
- 「推し」への愛を表現する
- [[AI時代のジョブ理論：感情的ジョブがデザイナーの武器]]

**社会的ジョブ：**
- 「Pok好き」としてのアイデンティティ表現
- センスの良さを示す（ギフト）
- コミュニティへの所属感

**ECサイト改善の示唆：**
- 機能的ジョブだけでなく、感情的・社会的ジョブも設計する
- 「推し」への愛を表現できる仕組み（マイページのカスタマイズなど）
- コミュニティ要素の検討

## 🔍 デザインリサーチ：調査の設計と実施

[[デザインリサーチの意義とプロセス]]

### ダブルダイアモンド

```
発見（Discover） → 定義（Define） → 開発（Develop） → 提供（Deliver）
  ◇              ◆             ◇              ◆
```

**現在のフェーズ：**
- まず「発見」フェーズで広くリサーチ（ユーザーインタビュー、行動観察）
- 次に「定義」フェーズでインサイトを抽出し、HMWを作成

### インタビュー設計

[[インタビュー技術の本質]]

**インタビューの3つのタイプ：**

1. **機会探索型**
   - Pokセンターオンラインの利用全体を俯瞰
   - 価値や未充足ニーズを見つける
   - → ペルソナ、価値仮説

2. **タスク分析型**
   - 購入プロセスの一連の行動・判断・困りごとを追う
   - → 要件抽出、改善ポイント

3. **仮説検証型**
   - 改善案の妥当性を検証
   - → 次のイテレーション

**質問設計の原則：**
- **具体から訊く**：抽象的な意見ではなく、具体的な行動や状況
- **時系列でたどる**：「最後に買い物をした時」から遡る
- **曖昧語を具体化させる**：「便利」「使いやすい」の中身を聞く

**インタビュー実施のコツ：**
- ラポール（信頼関係）の構築
- 沈黙を恐れない
- 「どう思ったか」を追う
- 矛盾に注目する

### インサイト抽出とHMW

[[インサイト抽出とHMW]]

**インサイトの3条件：**

1. **文章を読んで状況を理解できること**
   - ステークホルダーがパッと理解できる

2. **イノベーションのヒントとなっていること**
   - 行動を起こせるもの

3. **新規性があり、容易に予測できないこと**
   - 既知の情報ではない

**インサイトからHMWへ：**

> 「どうすれば私たちは**[対象となるユーザー]**のために**[制約]**を考慮しながら**[ゴール]**を提供できるだろうか」

**例：**
- インサイト：「ユーザーは推しPokの新商品を見逃したくないが、サイトを毎日チェックするのは面倒」
- HMW：「どうすればユーザーが推しPokの新商品情報を見逃さずに受け取れるか？」

## 🔗 サービスの裂け目：チャネル統合の重要性

[[サービスの裂け目：弱連結のマネジメント]]

### 裂け目の概念

> 「個々のサービスの連結が強ければ、すべてがうまくかみ合い1つの一貫したシームレスな体験になる。しかしそうではなく、ユーザーがサービスの『裂け目』に遭遇してしまうと、サービスの利用を継続できなくなったり、継続したいと思えなくなったりする」

**Pokセンターにおける裂け目のリスク：**

1. **オンラインと店舗の間**
   - オンラインで在庫ありと表示 → 店舗に行ったらない
   - 店舗でポイントカード → オンラインでは別システム
   - 店舗スタッフが知らないオンライン限定商品

2. **購入と配送の間**
   - 購入はスムーズ → 配送トラブルでコールセンターが対応できない
   - 注文履歴を見つけられない

3. **異なるチャネル間**
   - Webサイト、アプリ、店舗、SNSで情報が異なる
   - 用語の不一致（「予約」「受付」「申込」）

### 裂け目を防ぐ実践

**1. データの流れを可視化する**
- 誰がデータを収集し、誰がアクセス権を持っているか
- 同じデータを何度も収集していないか
- サービスブループリントの作成

**2. 最弱のチャネルが全体を決める**
- 最高のWebサイト × 最悪のコールセンター = 最悪の体験
- すべてのチャネルを80点に

**3. 一貫性のチェックリスト**
- [ ] すべてのチャネルで同じ用語を使っているか？
- [ ] データは共有されているか？
- [ ] プロセスは互換性があるか？
- [ ] ブランドボイスは一貫しているか？

**4. ステップの「間」をデザインする**
- 購入完了 → 発送 → 配達の間に何が起きるか
- ユーザーは何を期待しているか
- 待ち時間にどんな情報が必要か

## 🏪 店舗体験からECへの示唆

[[店舗体験デザインの示唆まとめ]]

### 示唆1：五感を使った「世界観への引き込み」

**店舗の強み：**
- 香り、視覚、音で世界観を伝える
- 入った瞬間の「Pokの世界」感

**ECへの適用：**
- ビジュアルデザインでの没入感
- アニメーション、サウンドの活用
- 「Pokの世界に入った」感をどう作るか

### 示唆2：「ここにいてもいい」という許可のデザイン

**店舗の強み：**
- スタッフの気配り
- サードプレイス的な心地よさ

**ECへの適用：**
- 急かさないUI（カウントダウンタイマーは慎重に）
- 「お気に入り」「後で見る」など、ゆっくり選べる仕組み
- 心理的安全性の演出

### 示唆3：価格の透明性が安心感を生む

**店舗の事例（はま寿司）：**
- 値段を常に可視化
- 「積み重なる焦り」を取り除く

**ECへの適用：**
- カート内の合計金額を常に表示
- 送料、手数料を早い段階で明示
- 「予想外の請求」への不安を取り除く

### 示唆4：UIは「焦りの文脈」を切り離す

**店舗の事例（スタバ）：**
- オーダー画面で落ち着いて選べる
- 列の焦りと違う文脈

**ECへの適用：**
- 商品比較機能（複数商品を並べて見られる）
- 「後で見る」リスト
- 検索中のローディングの見せ方

### 示唆5：「ハードルを下げる」ことで新市場を作る

**店舗の事例（チョコザップ）：**
- 徹底的に心理的・物理的ハードルを下げる

**ECへの適用：**
- 会員登録なしでも閲覧・カート追加可能
- ゲスト購入の容易さ
- 初回購入ハードルを下げる（送料無料、ポイント付与）

### 示唆6：コンセプトの「わかりやすさ」と「一貫性」

**店舗の事例：**
- 「何の店か」が3秒でわかる
- 入口 → 中 → 出口が一本線

**ECへの適用：**
- トップページで「何のサイトか」が即座にわかる
- ナビゲーションの一貫性
- トーン＆マナーの統一

### 示唆7：「エリアにおける希少性」→「目的における希少性」

**店舗の事例：**
- このエリアで唯一の〇〇

**ECへの適用：**
- Pokセンター限定商品
- オンライン限定商品
- 「ここでしか買えない」価値の明確化

## 💻 ECサイト特有の考慮点

### 1. 非消費という競合

[[非消費という競合：物欲がない人のジョブを見つける]]

> 「物欲がない＝ジョブがないではない。"買わない"こと自体が代替案のひとつ」

**物欲がない人の4タイプ：**

1. **無関心型**（ジョブ頻度が低い/痛みが弱い）
   - レバー：従量・レンタル、持たない解
   - PokEC：たまにしか買わない人へのアプローチ

2. **高負担回避型**（所有コスト過敏）
   - レバー：ゼロ負担設計、出口保証（下取り/買取/返品）
   - PokEC：「また買ってしまった」罪悪感への対応

3. **価値観一致重視型**（反消費・ミニマル・環境配慮）
   - レバー：長期耐用、修理容易性、静かな語り
   - PokEC：サステナビリティへの配慮

4. **選択疲れ/不安型**（情報過多・後悔回避）
   - レバー：1択化、お試し→自動継続、後悔保険
   - PokEC：膨大な商品数をどう整理するか

**負担予算（Burden Budget）：**
- 金銭的負担
- 時間的負担
- 認知的負担（選択、管理、心配事）
- 物理的負担
- 心理的負担（罪悪感）

### 2. 感情的ジョブの設計

[[AI時代のジョブ理論：感情的ジョブがデザイナーの武器]]

**GPT-4oの事例からの学び：**
- 機能的優位性だけでは勝てない
- 感情的ジョブを損なわないことが最重要
- マイクロインタラクションで感情を設計

**ECサイトへの適用：**

**完了時のマイクロインタラクション：**
- 「購入完了」→「素敵な選択ですね！」
- 「カートに追加」→「あなたのコレクションに追加しました」

**エラー時の言葉選び：**
- 「エラー」→「うまくいきませんでした。もう一度試してみましょう」
- 「在庫切れ」→「人気商品のため、現在品切れ中です。入荷お知らせを受け取りますか？」

**継続的な関係性：**
- パーソナライゼーション（推しPok設定）
- 購入履歴からのレコメンド
- 誕生日特典

### 3. 観察の技術

[[観察の技術と哲学：深く見る力の育て方]]

**競合サイトの観察：**
- AEIOUフレームワーク（Activities / Environments / Interactions / Objects / Users）
- 「なぜ？」を繰り返す
- 矛盾に注目する

**ユーザー行動の観察：**
- ヒートマップ、スクロール深度
- カート放棄率
- 離脱ポイント

**エポケー（判断保留）：**
- わかった気にならない
- 問いと仮説の無限ループ

## 🛠️ 実践のためのフレームワーク

### 1. ジョブインタビュー実施計画

**対象者：**
- [ ] ヘビーユーザー（月1回以上購入）
- [ ] ライトユーザー（年に数回購入）
- [ ] 離脱ユーザー（カート放棄、最近購入なし）
- [ ] 店舗のみ利用者（なぜオンラインを使わないか）
- [ ] 非消費者（Pok好きだけど買わない人）

**質問項目：**
1. 最後に購入したのはいつ？具体的な日付は？
2. そのとき何を達成しようとしていた？
3. どんな状況でそのジョブが立ち上がった？
4. 何が邪魔をしている？
5. 今はどのように対処している？
6. 理想的な解決策は？

**目標人数：**
- 各セグメント5-10人（合計30-50人）
- 「200人に聞くと勘所がわかる」が理想だが、まず30人から

### 2. カスタマージャーニーマップ作成

**フェーズ：**
1. **認知**：どこでPokセンターオンラインを知るか
2. **検討**：商品を探す、比較する
3. **購入**：カートに入れる、決済
4. **受取**：配送、開封
5. **使用/体験**：商品を楽しむ
6. **再購入/推奨**：リピート、口コミ

**各フェーズで記録：**
- [ ] 行動（何をするか）
- [ ] 思考（何を考えているか）
- [ ] 感情（どう感じているか）
- [ ] タッチポイント（どのチャネルを使っているか）
- [ ] ペインポイント（困っていること）
- [ ] 機会（改善できること）

### 3. サービスブループリント

**レイヤー：**
1. **Physical Evidence**：ユーザーが見るもの（UI、梱包、メール）
2. **Customer Actions**：ユーザーの行動
3. **Frontstage**：ユーザーに見える部分（Webサイト、カスタマーサポート）
4. **Backstage**：ユーザーに見えない部分（在庫管理、物流）
5. **Support Processes**：サポートシステム（決済システム、CRM）

**裂け目の特定：**
- どこでデータが途切れているか
- どこでチャネル間の連携が取れていないか

### 4. インサイト抽出ワークショップ

**ステップ：**
1. **切片化**：インタビュー内容を概念単位で切り出す
2. **グループ化**：類似概念を束ねる
3. **ラベル付け**：各グループに抽象化したラベル
4. **考察**：パターンから価値仮説や方向性を言語化
5. **HMW作成**：「どうすれば〇〇できるか？」

**チェックリスト：**
- [ ] 意見・提案は除外し、動機・感情・推論を残した
- [ ] チームで議論しながら進めた
- [ ] 「○○だからかもしれない」という考察を加えた
- [ ] リサーチに関わってこなかった人が理解できるか
- [ ] イノベーションのヒントになっているか
- [ ] 新規性があるか

### 5. 改善案の優先順位付け

**2軸で評価：**
- **縦軸**：インパクト（ジョブの達成度）
- **横軸**：実現容易性（工数、コスト、技術的難易度）

**優先度：**
1. **Quick Win**（高インパクト × 低工数）→ 最優先
2. **戦略的投資**（高インパクト × 高工数）→ 中長期
3. **改善**（低インパクト × 低工数）→ 余裕があれば
4. **延期**（低インパクト × 高工数）→ やらない

## 💡 重要な引用

> 「だれがでも何をでもなく、なぜ？がジョブ理論で追求すべきもの。」

> 「製品をビジネスのフレームに乗せて語っていたことがむしろ窮屈で、可能性を狭めていたのではないか？」

> 「個々のサービスの連結が強ければ、すべてがうまくかみ合い1つの一貫したシームレスな体験になる。」

> 「最高のWebサイトを作るより、コールセンターをマシにする。最先端のアプリを作るより、店舗との連携を考える。120点の体験を一つ作るより、すべてを80点にする。」

> 「観察のうまさがどのジャンルにおいても成功の要因となる」
