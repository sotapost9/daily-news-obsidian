#当事者研究 #PermanentNote

# インクルーシブデザインは最初から：外れ値の20%から始める

あるプロジェクトで、サービスの設計が終わってから「そういえばアクセシビリティは？」と聞かれたことがある。

「後から対応します」と答えたものの、実際には大変だった。既存のUIを全部見直し、コントラスト比を調整し、スクリーンリーダー対応を追加し……。結局、最初から考えていれば半分の工数で済んだはずの作業に、倍の時間がかかった。

『Good Service』に、Web開発者のコーデリア・マギー＝タブの言葉が引用されている。

> 「インクルージョンとは、ブルーベリーマフィンを作るようなもの。ブルーベリーは最初から入れておく方が、最後に入れるよりもずっと楽」

これを読んだとき、思わず笑ってしまった。確かに、焼き上がったマフィンにブルーベリーを後から押し込むのは無理がある。

インクルーシブデザインも同じだ。後から追加するものではなく、最初から組み込むべきものなのだ。

## 80/20の法則を逆転させる

ここからが面白い。『Good Service』はこう続ける。

> 「私たちは、容易に満たすことができる80％のニーズをデザインすることを考えるより、いわゆる外れ値的な20％のニーズを満たすことにもっと時間を費やすべきだ。そうすることで、最も困難な状況に置かれた人たちが、だれもが利用できるサービスへの道をどうすれば見つけることができるかを理解できるようになる。」

普通は、80%の「一般的な」ユーザーのために設計して、残り20%は後回しにする。でも、それを逆転させろ、と言っているのだ。

最初は「え？」と思った。20%のために設計したら、80%が犠牲になるんじゃないか、と。

でも、実際にやってみると、その逆だった。

### なぜ「最初から」なのか

**後から追加する場合の問題：**
1. **設計の根本的な見直しが必要**
   - 既存のアーキテクチャが対応できない
   - 大規模な改修が必要
   - コストが何倍にも膨れ上がる

2. **表面的な対応になりがち**
   - 「アクセシビリティチェッカーを通す」だけ
   - 本質的なインクルージョンにならない
   - むしろ不自然な体験を生む

3. **一貫性が保てない**
   - 一部のチャネルだけ対応
   - テスコ社の「Mx」事例の再来
   - [[サービスの裂け目：弱連結のマネジメント]]

**最初から組み込む場合のメリット：**
1. **自然に統合される**
2. **コストが抑えられる**
3. **より良い設計になる**
4. **全員が当事者意識を持つ**

## 📊 80/20の逆転：外れ値の20%から始める

[[智予-Good Service［グッド・サービス］　DX時代における"本当に使いやすい"サービス作りの原則15]]

> 「私たちは、容易に満たすことができる80％のニーズをデザインすることを考えるより、いわゆる外れ値的な20％のニーズを満たすことにもっと時間を費やすべきだ。そうすることで、最も困難な状況に置かれた人たちが、だれもが利用できるサービスへの道をどうすれば見つけることができるかを理解できるようになる。」

### 従来のアプローチ（80%から始める）

**典型的な思考：**
- まず「普通の人」のために設計する
- 80%のユーザーに最適化する
- 残り20%は後で考える（実際には考えられない）

**問題点：**
1. **「普通」の定義が偏っている**
   - 設計者の視点での「普通」
   - マジョリティの基準
   - 多様性を見落とす

2. **外れ値が永遠に取り残される**
   - 後で対応する予算は出ない
   - 技術的負債が蓄積
   - 社会的排除を助長

3. **イノベーションの機会を逃す**
   - 制約が創造性を生むことを知らない
   - 外れ値の洞察が80%にも価値をもたらす可能性

### 新しいアプローチ（20%から始める）

**逆転の発想：**
- 最も困難な状況の人から設計する
- 外れ値のニーズを満たす
- 結果として80%もカバーできる

**なぜ機能するのか：**

1. **制約が良い設計を生む**
   - シンプルになる
   - わかりやすくなる
   - 誰にとっても使いやすくなる

2. **本質的な課題が見える**
   - 表面的な問題ではなく根本的な問題
   - 「当たり前」を疑える
   - 構造的な改善につながる

3. **潜在的なペインを解消**
   - 80%の人も薄々感じていた使いづらさ
   - 言語化されていなかった不便さ
   - 「こっちの方が良い」という発見

## グッドグリップスのピーラーが教えてくれたこと

料理好きの人なら、グッドグリップスのピーラーを見たことがあるかもしれない。太くて黒いハンドルが特徴的な、あのピーラーだ。

あのピーラー、実は関節炎の患者のために作られた。握力が弱い人でも使えるように、ハンドルを太く、握りやすくした。言わば「マイノリティのためのデザイン」だ。

ところが、発売してみると、健常者からも「こっちの方が使いやすい」と評判になった。今では、ユニバーサルデザインの代表例として語られている。

なぜこんなことが起きたのか。

関節炎患者という「N=1」から始めたことで、「握力」という本質的な課題に向き合うことになった。そして、その課題を解決したら、実は多くの人が潜在的に感じていた「ピーラーって握りにくいよね」という不満も、一緒に解消されていたのだ。

マイノリティのための解決が、マジョリティにも価値をもたらす。これが「外れ値の20%から始める」ことの威力だ。

以前読んだnote記事に、こんな一節があった。

> 「繊細な感覚を持った人から起点にしたことで、みんなが潜在的に少し感じていた使いづらさも解消するに至った」

まさにそういうことなのだと思う。

## 「普通」の罠：Google Photoが犯した過ち

2015年、ソフトウェアエンジニアのジャッキー・アルシネが、Google Photoの写真自動タグ付け機能で、衝撃的なことを発見した。

彼の友人たち（全員黒人）の写真に、AIが「ゴリラ」とタグをつけていたのだ。

彼がこのことをTwitterに投稿すると、大炎上した。Google社は即座に謝罪し、「ゴリラ」タグを削除した。でも、問題の本質はそこじゃない。

『Good Service』はこう指摘する。

> 「彼らは、自分たちが開発したプロダクトを、白人のみに──たぶん、自分たちと同じ野生動物愛に溢れる白人のみに──しかテストしなかったのだ。」

開発チームのほとんどが白人だった。テストも、自分たちと似た人だけで行った。有色人種の視点は、最初から入っていなかった。

悪意はなかったと思う。でも、結果として、構造的な差別を生んでしまった。

「差別は大抵悪意のない人がする」という本があるらしいが、まさにその通りだ。私たちが「普通」だと思っているものは、実は私たち自身の偏見を反映しているだけかもしれない。

『Good Service』はさらに厳しいことを言う。

> 「私たちにとっての『普通』は、それを定義する人の偏見に大きく左右される。残念なことに、多くのサービス組織は多様性に欠けているため、『普通』とは『若くて、白人で、男性の、シスジェンダーである異性愛者』と定義するサービスもたくさんある。」

だからこそ、最初から多様性を組み込む必要がある。後から「あ、マイノリティのこと忘れてた」では遅い。

## 盲学校の経営者が学んだこと

もう一つ、『Good Service』に載っていた印象的な事例がある。

19世紀のアメリカに、ハウという盲学校の経営者がいた。彼は真剣に視覚障害者の教育を考え、「浮き出し文字」という新しい文字システムを開発した。晴眼者にも読めるし、盲人も触って読める。画期的なアイデアだった。

しかし、生徒たちは点字を選んだ。

点字は晴眼者には読めない。でも、盲人にはこちらの方がずっと読みやすかった。ハウの「浮き出し文字」は、結局、教師の都合を優先したシステムだったのだ。

> 「ハウの生徒たちは、次第に（晴眼者の教師たちには読むことが困難な）点字で手紙を交換し合うようになり、授業の作文や宿題も点字で書くようになった。最終的に、ハウは負けを認めざるをえなかった。結局彼は、自分とは異なる人たちにとって何が一番よいかということをわかっていなかったのだ。」

ハウには悪意がなかった。むしろ、善意しかなかった。でも、当事者の声を聞いていなかった。

「あなたのために」という善意は、時に傲慢になる。専門家が「何が良いか」を決めてしまう。でも、本当に必要なものは、当事者にしかわからない。

障害者支援の世界に、「Nothing About Us Without Us（私たちのことを、私たち抜きに決めないで）」という言葉がある。まさにこのことだ。

## 🎯 インクルーシブデザインの実践原則

### 1. 当事者を共同研究者にする

[[インクルーシブリサーチ]]
[[弱さの共有が生む強さ：当事者研究と組織論]]

**Nothing About Us Without Us：**
- 当事者抜きに決めない
- 対象ではなく主体
- 対等な関係性

**「ともに弱くなる」：**
> 「当事者研究は、統合失調症を抱える若者とソーシャルワーカーの対話から始まった。『ともに弱くなる』というプロセスが出発点であり、従来の心理教育プログラムとは異なる。」

**実践方法：**
- 当事者ワークショップ
- 共同デザインセッション
- 継続的な対話
- 報酬の支払い（搾取にならないように）

### 2. チームの多様性を確保する

**多様性のチェックリスト：**
- [ ] 人種・民族の多様性
- [ ] 性別・性自認の多様性
- [ ] 年齢の多様性
- [ ] 障害の有無の多様性
- [ ] 社会経済的背景の多様性
- [ ] 地域・文化の多様性
- [ ] 家族構成の多様性

**なぜ多様性が必要か：**
- 無意識の偏見を相殺する
- 多様な視点から問題を見る
- 「普通」の定義を問い直す

### 3. エッジケースを中心に設計する

**従来のアプローチ：**
```
ハッピーパス → エッジケース → エラーハンドリング
（メイン）      （サブ）        （おまけ）
```

**インクルーシブアプローチ：**
```
エッジケース → ハッピーパス → 全体の最適化
（起点）       （結果）       （統合）
```

**具体例：**

**従来：**
1. 健常者向けに設計
2. 車椅子ユーザーのために「別ルート」
3. 結果：バリアフリーが不自然

**インクルーシブ：**
1. 車椅子ユーザーを起点に設計
2. 段差のない動線
3. 結果：ベビーカー、高齢者、すべてに優しい

### 4. 制約を創造の源泉にする

**制約から生まれる創造性：**
- シンプルさ
- 明確さ
- 本質的な改善

**実例：**
- 字幕 → 音が出せない環境でも視聴可能に
- 音声入力 → 運転中の操作が可能に
- ダークモード → 視覚障害者だけでなく、夜間使用者に

## 🚧 よくある障害：ユーザーができないこと

[[智予-Good Service［グッド・サービス］　DX時代における"本当に使いやすい"サービス作りの原則15]]

> 「サービス内に存在する障害が何かを知るためにはユーザー調査をする必要があるが、一般的にユーザーにとって障害になるものをここに紹介する。」

### 認知的な障害

**長い番号を記憶する**
- 読字障害（ディスレクシア）
- 協調運動障害
- その他の認知機能障害

**複雑な指示に従う**
- 自閉症
- 学習障害
- 注意欠陥

**日付や時間、予約などを記憶する**
- 記憶障害
- 多忙な生活
- ストレス

### 物理的な障害

**物理的にどこかへ移動する**
- 身体的な障害
- 社交不安障害
- 車を持っていない
- 公共交通機関を利用できない
- 交通費がない

**就業時間内にどこかへ行ったり何かをしたりする**
- 標準的な勤務時間で働く人
- シフト勤務
- 子育て中

### 設計上の配慮

**すべきこと：**
- 代替手段を必ず用意する
- 必要条件を最小限に抑える
- ユーザーの行動を促すようデザインする
- アクセス不可能なものがある場合の対策

**してはいけないこと：**
- 「普通の人ならできる」という前提
- 単一の方法しか提供しない
- 外れ値を無視する

## 🌍 インクルージョンと平等

[[智予-Good Service［グッド・サービス］　DX時代における"本当に使いやすい"サービス作りの原則15]]

> 「また、『ユーザーが、他のどの潜在的ユーザーとも平等に扱われることを実感しながらサービスを利用できるようにする』ことも、インクルージョンの重要な目的と言える。」

### インクルージョンの2つの側面

**1. 実際にアクセスできること**
- 物理的なアクセス
- 認知的なアクセス
- 経済的なアクセス
- 時間的なアクセス

**2. 平等に扱われていると実感すること**
- 心理的安全性
- 尊重されている感覚
- 「特別扱い」ではなく「当然の権利」
- [[弱さの共有が生む強さ：当事者研究と組織論]]

### 当事者以外の心理的安全性

> 「当事者以外の心理的安全性も高められのがインクルージョン」

**インクルーシブな環境の効果：**
- マイノリティが安心できる
- マジョリティも「間違えても大丈夫」と思える
- 全員が学び、成長できる
- 対立ではなく共創

## 📋 インクルーシブデザインのチェックリスト

### 設計段階

- [ ] 当事者を共同研究者に含めているか？
- [ ] チームは多様性があるか？
- [ ] エッジケースから設計を始めているか？
- [ ] 「普通」の定義を問い直したか？
- [ ] 複数の代替手段を用意しているか？

### テスト段階

- [ ] 多様なユーザーでテストしているか？
- [ ] エッジケースでテストしているか？
- [ ] 様々な状況（時間帯、場所、デバイス）でテストしているか？
- [ ] 障害のある人でテストしているか？
- [ ] 異なる文化背景の人でテストしているか？

### 実装段階

- [ ] すべてのチャネルで一貫しているか？
- [ ] スタッフはトレーニングを受けているか？
- [ ] ドキュメントは包摂的か？
- [ ] フィードバックループがあるか？
- [ ] 継続的な改善プロセスがあるか？

### 評価段階

- [ ] インクルージョンのKPIを設定しているか？
- [ ] 外れ値のユーザーの満足度を測定しているか？
- [ ] 離脱率を分析しているか（どこで誰が離脱するか）？
- [ ] 当事者からのフィードバックを定期的に得ているか？

## 最後に：ブルーベリーを最初から入れる勇気

インクルーシブデザインは、正直言って面倒だ。

「後でやります」と言いたくなる。でも、それはマフィンが焼き上がってからブルーベリーを押し込むようなものだ。

最初から、外れ値の20%のために設計する。最初から、当事者を共同研究者にする。最初から、多様性を組み込む。

面倒だけど、結局はその方が楽だし、良いものができる。

そして何より、グッドグリップスのピーラーみたいに、マイノリティのための解決が、マジョリティにも価値をもたらすことがある。

あなたのプロジェクトで、「外れ値の20%」は誰だろう？その人たちの声を、最初から聞いているだろうか？

ブルーベリーを最初から入れる勇気。それが、本当に良いサービスを作る第一歩なのかもしれない。
